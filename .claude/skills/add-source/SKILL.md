# Quelle analysieren & SecondBrain erweitern

Zweistufiger Workflow für dieses Repo: rohes Analysematerial landet unter
`/sources`, daraus entsteht eine kuratierte Seite unter `/site/docs-<bereich>/`.
Der Gemini-Extraktionsschritt (Prompt: `sources/prompt-templates/gemini-video-analysis-prompting.md`)
bleibt manuell — der User liefert das Gemini-Rohoutput im Chat.

## Ablauf

### 1. Quelltyp bestimmen

- **YouTube-Video**: User liefert Gemini-Rohoutput (Transkript-Auszug,
  Screens, Struktur, Begriffe) im Chat.
- **Dokumentation/Artikel**: User liefert einen Link oder Text einer externen
  Doku/Quelle direkt (kein Gemini-Zwischenschritt nötig).

### 2. Zielbereich & Unterordner per Interview klären

**Nicht selbst entscheiden** — immer nachfragen, auch wenn ein Thema
naheliegend erscheint:

1. Vorhandene Dokubereiche zeigen (`ls site/docs-*`) und fragen, ob die
   Quelle in einen bestehenden Bereich gehört oder ein neuer Bereich nötig
   ist.
   - **Neuer Dokubereich** = eigene `plugin-content-docs`-Instanz in
     `docusaurus.config.js` (neues Plugin, neuer Sidebar, neuer
     Navbar-Eintrag, neuer Footer-Eintrag) **plus** eine neue Kachel in
     `site/src/components/HomepageFeatures/index.js` (`DocAreas`-Array,
     sonst ist der Bereich vom Dashboard aus nicht erreichbar) — das ist
     eine strukturelle Änderung, die **nicht** im selben Zug mitgemacht
     wird. Stattdessen: kurz mit dem User abstimmen, dass dafür erst die
     Config erweitert wird, bevor Inhalte entstehen.
2. Innerhalb des Bereichs vorhandene Unterordner zeigen (z. B.
   `site/docs-ki-entwicklung/methoden`) und fragen, ob die Quelle dorthin
   gehört oder ein neuer Unterordner (mit eigener `_category_.json`) sinnvoll
   ist.
3. Passenden Dateinamen/Slug mit dem User kurz bestätigen (kebab-case,
   sprechend, z. B. `karpathy-claude-spec-verifier-environment`).

### 3. Rohe Zusammenfassung unter `/sources` ablegen

- Alle Quellenarten (YouTube wie externe Doku/Artikel) landen einheitlich
  unter `sources/analysis/<slug>.md`.
- YouTube-Quelle: Struktur aus
  `sources/prompt-templates/claude-video-analysis-create-doc.md` verwenden.
- Externe Doku/Artikel: gleiche Grundstruktur (Quelle, Thema, Datum,
  Zusammenfassung, Kernkonzepte, ggf. Prompt-Vorlagen/Code, Schritt-für-Schritt,
  Tools & Ressourcen, Notizen-Platzhalter) verwenden.
- Diese Datei ist die **vollständige Rohanalyse** — nichts kürzen, was im
  Gemini-Output bzw. der Quelle stand.

### 4. Docusaurus-Seite erstellen

- Datei unter `site/docs-<bereich>/<unterordner>/<slug>.md`, im Stil von
  `site/docs-ki-entwicklung/methoden/spec-verifier-environment.md`:
  - Front-Matter mit `sidebar_position` (an existierende Geschwister-Dateien
    anpassen, per `_category_.json` bzw. vorhandene Dateien prüfen).
  - `:::info[Quelle]` ... `:::`-Block (Docusaurus-v3-Syntax, Titel in
    eckigen Klammern) mit Link zur Originalquelle und Link zur
    Rohanalyse-Datei in `sources/` (GitHub-Blob-URL,
    `https://github.com/Tecch-Git/SecondBrain/blob/main/...`).
  - Inhalt ist eine **kuratierte, gekürzte** Fassung der Rohanalyse (keine
    1:1-Kopie) — fürs schnelle Nachlesen/Wiedererinnern optimiert.
- Sidebars sind in diesem Repo `autogenerated` (siehe
  `site/sidebars-ki-entwicklung.js`) — keine manuelle Sidebar-Pflege nötig.
  Bei neuem Unterordner ggf. eine `_category_.json` nach Vorbild von
  `site/docs-ki-entwicklung/methoden/_category_.json` anlegen.

### 5. `intro.md` des Dokubereichs aktualisieren

- Jeder Dokubereich hat eine handgepflegte `site/docs-<bereich>/intro.md` mit
  einer Kapitel-Liste (Methoden, Skills, ggf. Unterkategorien). Diese Liste
  ist **kein Autogenerated-Sidebar-Ersatz**, sondern der Einstiegspunkt beim
  Klick auf den Navbar-Eintrag — sie muss bei jeder neuen Quelle mitgepflegt
  werden, sonst veraltet sie.
- Prüfen, ob die Quelle in eine bereits gelistete Kategorie fällt:
  - **Bestehende Kategorie, neue Seite darin** (z. B. neue Datei in
    `skills/copilot/`): den Link auf die neue Doku-Seite unter der
    passenden Kategorie ergänzen (ggf. zusätzlich zu vorhandenen Links in
    dieser Kategorie, nicht ersetzen).
  - **Neue Unterkategorie** (z. B. neuer Unterordner mit eigener
    `_category_.json`, etwa `skills/<neues-tool>/`): einen neuen
    Unterpunkt mit Link auf die Kategorie-Übersichtsseite
    (`/docs/<routeBasePath>/category/<slug>`) plus Link auf die erste
    Doku-Seite darin ergänzen.
  - **Neue Top-Level-Kategorie** (eigener Ordner mit `_category_.json`
    direkt unter `docs-<bereich>/`): einen neuen Top-Level-Listenpunkt mit
    Link auf die Kategorie-Übersichtsseite ergänzen.
- Link-Konventionen (siehe `site/docs-ki-entwicklung/intro.md` als Vorbild):
  - Links auf konkrete Doku-Seiten: relativer Markdown-Link
    (`./<unterordner>/<slug>.md`).
  - Links auf Kategorie-Übersichtsseiten (generated-index, keine echte
    Datei): absoluter Pfad `/docs/<routeBasePath>/category/<kategorie-slug>`
    (z. B. `/docs/ki-entwicklung/category/skills`) — **nicht** mit
    abschließendem `/` auf den Ordnerpfad selbst verlinken (`./skills/`),
    das führt zu einem Broken-Link-Build-Fehler, da die echte Route
    `/category/<slug>` lautet, nicht der Ordnerpfad.
- Nach der Änderung erneut `npm run build` laufen lassen (Teil des
  Pflicht-Build-Checks aus Schritt 6) — der Broken-Link-Checker
  (`onBrokenLinks: 'throw'`) deckt falsche Kategorie-Links sofort auf.

### 6. Branch wählen, committen, PR erstellen

Diesen Schritt führt der geteilte Ablauf aus
`.claude/skills/repo-publish-flow/SKILL.md` aus (Sync mit Remote,
Branch-Entscheidung, Build-Check, Commit, Push, PR) — diese Datei lesen und
befolgen. Dafür liefern:

- **Dateien:** Rohanalyse, Doku-Seite, ggf. `_category_.json`, und die
  aktualisierte `intro.md` aus Schritt 5.
- **Branch-Namens-Vorschlag:** `claude/add-source-<slug>`.
- **PR-Titel:** `Add <Kurzthema>`.
- **Summary-Bullets:**
  - Rohanalyse `<Quelle/Thema>` unter `sources/analysis/<slug>.md`
  - Doku-Seite unter `site/docs-<bereich>/<unterordner>/<slug>.md`
- **Test-Plan-Bullets:** Standard aus `repo-publish-flow` (Build + Preview-Check).

Bei "bestehender Branch/PR passt thematisch" zusätzlich zu `repo-publish-flow`
Schritt 2: anhand des `add-source`-PR-Titels/-Beschreibung prüfen, ob die neue
Quelle dazupasst.

## Nicht tun

- Keine Inhalte erfinden, die nicht in der Gemini-Rohanalyse/Quelle stehen.
- Keinen neuen Dokubereich (= neue Plugin-Instanz) ohne explizite Bestätigung
  anlegen.
- Keine Sidebar-Dateien händisch pflegen (autogenerated).
- Branch-Wahl (neu vs. bestehend) nicht selbst entscheiden, nur empfehlen.
- Nicht automatisch mergen — PR erstellen/aktualisieren ist das Ende des
  Ablaufs.
- `intro.md` des betroffenen Dokubereichs nicht vergessen/auslassen — sie
  veraltet sonst stillschweigend (siehe Schritt 5).
